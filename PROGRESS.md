다음에 이 프로젝트를 이어서 작업할 때(사람이든 Claude든) 여기부터 읽을 것.
`README.md`는 사용법 중심, 이 파일은 "왜 이렇게 만들었는지"와 지금까지의 경위 중심.

## 프로젝트 개요

- 목적: 한국투자증권(KIS) Developers API로 붙는 실제/모의 주식 트레이딩 봇 (C++ 초안)
- 위치: `C:\Users\wogur\OneDrive\Desktop\project\trading`
- GitHub: https://github.com/RedStones-112/trading-bot — **public repo**
- 빌드: CMake + MSYS2 UCRT64 g++. WinHTTP(Windows 내장)로 HTTPS, libcurl 등 외부
  의존성 없음. JSON은 nlohmann/json 헤더 하나만 vendor(`third_party/json.hpp`).

## 아키텍처

- `src/broker.hpp`: `IBroker` 인터페이스 + `StockInfo`(code/name/price/dayChangePct).
  구현체 3개:
  - `KisClient` (src/kis_client.*): 실제 KIS REST API
  - `MockBroker` (src/mock_broker.*): 완전 오프라인, 가상 종목 5개 랜덤워크
  - `SimBroker` (src/sim_broker.*): KIS 실시세는 그대로 쓰되 주문은 로컬 가상 체결
- `src/strategy.hpp`: 순수함수 3개 — `smaCrossSignal`(골든/데드크로스), `smaMomentum`,
  `netProfitPct`(수수료/세금 반영 순손익률)
- `src/news_crawler.hpp/cpp`: RSS 피드 파싱(직접 짠 `<item>`/`<title>` 추출, XML
  라이브러리 없음) + Google News 검색(`news.google.com/rss/search`) + 키워드 기반
  감성 점수화
- `src/main.cpp`: config.json 로드 → mode별 브로커 생성 → 정규장 체크 → (보유 없으면)
  스캔/선택/매수, (보유 중이면) 모니터링/매도 루프. `holdings.txt`를 5초마다 갱신하는
  별도 스레드도 여기서 띄움.
- `tests/test_strategy.cpp`: 결정론적 self-check (SMA 크로스, netProfitPct,
  MockBroker, NewsCrawler 감성점수). `cmake --build build --target test_strategy`로 빌드.

## 동작 흐름 (현재 최종본)

0. `paper`/`live` 모드는 평일 09:00~15:30(로컬 시각)가 아니면 스캔 자체를 스킵 (장
   마감 시 KIS가 주문을 거부하는데, 그 전에 스캔+뉴스크롤링을 다 하는 낭비를 막기 위함).
1. 보유 종목 없으면: KIS 거래량순위 API(세그먼트 `0000`/`1001`/`2001` 병합, 세그먼트당
   최대 30개 → 중복 제거 후 최대 ~90개) → 레버리지/인버스 ETF·ETN 이름 필터링 →
   `watchlist_size`만큼 후보 확보.
2. 후보가 `deep_scan_limit`보다 많으면 전일대비율(`prdy_ctrt`, 순위조회 응답에 이미
   있는 무료 필드)로 정렬해서 상위 N개만 정밀 분석(일봉 히스토리 조회는 비용이 큼).
3. RSS 뉴스(매경/한경/연합뉴스) 크롤링 → 종목명 언급 헤드라인에서 긍정/부정 키워드로
   감성 점수화.
4. 각 후보의 SMA 골든크로스 여부 확인 → Buy 신호 후보 중 기댓값(`이득 × 확률`, 이득은
   `take_profit_pct` 기준 목표수익, 확률은 감성점수를 0.05~0.95로 매핑한 휴리스틱)이
   가장 높은 종목 **하나만** 매수.
5. 보유 중엔 매 poll마다: (a) 데드크로스, (b) 수수료/세금 차감 순손익률이
   `take_profit_pct` 이상, (c) 보유종목명으로 Google News를 직접 검색한 감성점수가
   `bad_news_sentiment_threshold` 이하(악재) — 셋 중 하나만 걸려도 즉시 매도.
   (c)는 보유종목이 스캔의 거래량순위 후보에서 벗어나도 계속 추적하기 위해 스캔의
   뉴스풀과 별개로 매번 직접 검색함.
6. `holdings.txt`(5초 고정 간격) / `trades.log`(체결마다 CSV 한 줄) / 콘솔+`trading.log`
   (실시간 스캔/시그널/체결 로그).

## 사용자가 확인해준 설계 결정

- 기댓값 공식: **이득 × 확률**(표준). "이득/확률"은 채택 안 함.
- 매도 트리거: 데드크로스 **and** 수익실현 **and** 악재감지, 셋 다 사용(OR 조건).
- 뉴스 크롤링 범위: **RSS만** 우선 구현. 유튜브/SNS는 명시적으로 제외
  (공식 API 인증 필요, 비공식 스크래핑은 ToS/차단 위험 커서 이 단계에서는 안 함).
- GitHub repo는 처음 private로 만들었다가 사용자 요청으로 public 전환.

## 발견하고 고친 버그들 (시간순)

1. WinHTTP `GetLastError()`를 핸들 정리(`WinHttpCloseHandle`) *이후*에 호출해서 항상 0으로
   나오던 버그 — 실패 직후에 캡처하도록 수정.
2. WinHTTP 연결 포트를 443으로 하드코딩 — KIS는 모의투자 29443, 실서버 9443 커스텀
   포트를 씀. `http::request`에 포트 파라미터 추가.
3. 콘솔에 한글이 깨져 보임 — `SetConsoleOutputCP(CP_UTF8)` 누락. (로그 파일 자체는
   원래도 정상이었음, 콘솔 표시만 문제)
4. KIS 레이트리밋(`EGW00201`, 모의투자 계좌가 특히 엄격) — 시작 시 여러 API 호출이
   연달아 나가서 발생. 호출 사이 1.1초 pause 추가 + 일봉 히스토리는 매수 시점에
   한 번만 캐시(매 poll마다 재조회 안 함)로 호출 수 자체를 줄임.
5. `watchlist_size=100`을 줘도 10개만 반환 — 거래량순위 API가 세그먼트 하나당 최대
   30개까지만 응답하는 게 원인. 세그먼트 3개(`0000`/`1001`/`2001`) 병합+중복제거로
   해결(최대 ~90개까지 확보 가능).
6. MockBroker 회귀 버그: 순위조회 최적화(현재가를 랭킹 응답에서 재사용) 이후, mock
   모드의 랜덤워크가 `getCurrentPrice()`에서만 진행되고 `getTopVolumeStocks()`에서는
   안 움직여서 스캔 중 가격이 고정돼 있던 문제 — `advance()` 헬퍼로 공유해서 해결.

## 다중 포지션 포트폴리오로 확장 (2026-07-23)

기존엔 종목 하나만, 고정 `qty`(=1주)로만 보유했음. 사용자 요청으로 4가지 규칙 추가:
잔고 기반 운용, 여유자금 있으면 추가매수, 더 나은 후보 나오면 최저 기댓값 보유종목과
교체, 낱개매수 지양. 구현 전 Plan 서브에이전트로 설계 검증을 한 차례 거쳤고, 그
과정에서 아래 버그/설계 실수를 미리 잡았음 — 나중에 이 로직을 건드릴 때 왜 이렇게
됐는지 참고할 것.

- **로테이션 부등식 방향**: 처음 설계는 `신규후보EV > 보유종목EV - 교체비용`이었는데,
  이러면 비용이 클수록 오히려 교체가 쉬워지는 반대 방향 버그였음. 올바른 조건은
  `신규후보EV > 보유종목EV + 교체비용` — "교체가 그 비용을 감수할 만큼 나아야 한다"는
  뜻이 되려면 비용을 후보 쪽 문턱을 높이는 데 써야 함. 교체 대상(최저 EV 보유종목)
  선정 자체는 비용 반영 없이 순수 EV 최솟값으로 고름(비용을 선정에도 섞으면 비싼
  포지션이 부당하게 안 팔리는 쪽으로 편향됨).
- **평단가에 수수료 섞지 않기**: 가중평균 평단가(`avgBuyPrice`) 계산은 반드시
  체결가(fee 안 붙은 순수 가격)로 해야 함. `netProfitPct`가 fee/tax를 이미 별도로
  반영하는데 평단가에도 fee가 섞이면 익절 판정이 이중으로 틀어짐.
- **매수 수량**: `position_cash_fraction × 그 시점 매수가능금액`으로 계산하되, 그
  결과가 0주여도 실제 매수가능금액으로 1주는 살 수 있으면(비싼 종목) 1주로 폴백함 —
  안 그러면 낱개매수 지양 규칙 때문에 오히려 살 수 있는 매수 기회를 놓치는 역설이
  생김.
- **락 처리**: 스캔/시세/뉴스 조회 같은 느린 I/O는 항상 언락 상태에서 하고, 공유
  포지션 맵을 실제로 바꿀 때만 짧게 락. 매수/매도 실행 스레드는 메인 루프 하나뿐이라
  (statusWriterLoop는 읽기 전용) 동시 mutation 경합 자체는 없음.
- **broker에 fee/tax를 상태로 안 둠**: MockBroker/SimBroker가 가상 현금 잔고를
  정확히 증감하려면 fee_rate/tax_rate가 필요한데, 생성자에 박아넣지 않고
  `placeMarketOrder(..., feeRate, taxRate)` 호출 시점 인자로 받음 — KisClient는 그냥
  무시(실계좌 잔고에 이미 반영돼 있으므로).
- **`getBuyableCash()` 실제 검증**: KIS 매수가능조회(`inquire-psbl-order`,
  `ord_psbl_cash` 필드)로 구현하고, 실제 모의투자 서버에 붙여서 ~994만원(기본 1000만원
  계좌 기준)이 나오는 것까지 확인함. `mode: "live"`(실전 서버)로는 아직 검증 안 함.
- mock 모드로 짧게 돌려서 확인: 한 사이클에 8주 매수(고정 1주 아님, 규칙 4 확인),
  두 종목 동시 보유(규칙 2 확인), 같은 종목 추가매수 시 가중평균 평단가가 정확히
  계산됨(14주@60294 + 7주@60182 → 21주@60257 확인)까지 검증. 로테이션(규칙 3)은
  이 짧은 검증 실행에서는 조건이 실제로 발동하지 않아서 라이브로 직접 보진 못했고
  코드 리뷰로만 검증함 — 나중에 `max_positions`를 낮게(예: 1) 잡고 오래 돌리면
  더 쉽게 관찰 가능할 것.

## 한 종목에 계속 쪼개서 추가매수되던 문제 (2026-07-23)

실계좌 paper 모드로 돌리다가 세아메카닉스(396300) 한 종목에 728→511→358→...→10주
식으로 9분 내내 계속 추가매수만 되는 걸 발견. `trading.log`로 원인 확인:

- 그 종목 현재가가 9분 내내 정확히 4095원으로 고정(사실상 거래가 멈춘 종목으로
  보임). `smaCrossSignal`은 "저장된 과거 일봉 + 지금 이 순간의 실시간가" 두 값으로
  매 poll마다 새로 크로스 여부를 판정하는 구조라서, 가격이 크로스 임계값 위에
  머물러 있는 한(이 경우엔 가격이 아예 고정이라 계속) **매 poll 계속 BUY로 나옴** —
  "막 크로스한 순간"에만 한 번 잡히는 게 아니라 크로스 이후 계속 위에 있는 동안
  내내 잡힘. 원래(단일 포지션 시절) 코드는 보유 중이면 스캔 자체를 안 했어서 이
  문제가 안 보였는데, 이번 다중 포지션 개편으로 보유 중에도 매 poll 스캔하게 되면서
  드러남.
- 기댓값도 가격이 고정이라 매번 40.95로 동일 → 항상 최고 기댓값 후보로 뽑힘 →
  이미 보유 중이니 매번 "추가매수" 분기 → `position_cash_fraction`(0.3)이 "그 시점
  남은 현금"에만 적용되니 728, 511(×0.7), 358(×0.7)... 식으로 기하급수적으로 줄어드는
  분할매수가 반복됨.

**사용자 선택으로 신호 로직(smaCrossSignal 재계산 방식)은 그대로 두고, 종목당 상한
두 개로 막음**(신호가 계속 BUY로 남아있는 것 자체는 "버그"가 아니라 이 구현의
특성으로 두기로 함):
- `max_topups_per_symbol`(기본 3) — 한 종목에 허용하는 최대 추가매수 횟수(최초
  매수 제외). Position에 `topUps` 카운터 추가, 상한 도달 시 그 종목은 더 이상
  추가매수 안 함.
- `max_position_value_fraction`(기본 0.5) — 한 종목의 총 투입액(원가 기준
  `avgBuyPrice × qty`)이 전체 계좌 가치(매수가능금액 + 모든 보유종목 시가평가액)의
  이 비율을 넘지 않도록, 넘기 직전이면 매수 수량을 그만큼만 줄여서(0으로 막는 게
  아니라 딱 남는 여유만큼만) 매수함.
- 둘 다 "그 사이클엔 그 종목 매수 안 함"까지만 하고, 다른 종목으로 자동 전환하진
  않음(Phase B가 사이클당 "최고 기댓값 후보 1개"만 계산하는 기존 단순 구조를
  유지하기로 함 — 후보 랭킹 전체를 들고 다니면서 다음 순위로 폴백하는 건 이번
  범위 밖으로 둠. 실제 시장에서는 가격이 이렇게 오래 고정인 경우가 드물어서, 다음
  스캔에서 다른 종목이 최고 기댓값이 되면 자연히 그쪽으로 넘어갈 것으로 봄).

## 주문 접수 ≠ 체결 -- 전량 미체결이었던 사고 (2026-07-23)

세아메카닉스 무한분할매수(위 항목)를 상한으로 막은 뒤, 사용자가 "거래내역에 아무것도
없다"고 지적. 확인해보니 **더 근본적인 문제**였음: KIS 주식일별주문체결조회
(`inquire-daily-ccld`)로 그날 실제 체결 내역을 직접 조회하니

```
"tot_ord_qty": "2441"   <- 주문 수량 합계 (trades.log와 일치)
"tot_ccld_qty": "0"     <- 실제 체결 수량 = 0
```

**24건 주문 전부 접수만 되고 단 1주도 체결이 안 됐음.** `placeMarketOrder`가 KIS
주문 API의 `rt_cd=="0"`(접수 성공)만 확인하고 바로 "체결"로 간주해서 포지션/
`trades.log`에 반영하던 게 원인 — 세아메카닉스가 9분 내내 가격이 안 움직인 것도
사실 이 종목이 그 시점에 실체결이 거의 없는 상태였다는 신호였던 것으로 보임. 참고로
`매수가능금액`(`getBuyableCash`)이 계속 낮게 나왔던 것도 실제로 돈이 나간 게 아니라
KIS가 미체결 주문분만큼 매수가능금액에서 예약(잠금)해뒀기 때문 — 예수금
(`inquire-balance`의 `dnca_tot_amt`)은 처음부터 그대로 1,000만원이었음.

**시도했다가 버린 방법**: 처음엔 KIS 주식일별주문체결조회를 특정 주문번호(`ODNO`)로
필터링해서 그 주문 하나의 체결 여부를 직접 조회하려 했는데, 실제로 호출해보니
`ODNO` 파라미터를 넣어도 그날 전체 합계(`output2`)만 나오고 `output1`(개별 주문 내역)은
항상 비어있어서 특정 주문의 체결 여부를 이 엔드포인트로는 구분할 수 없었음(모의투자
특성일 수도 있음, 문서와 다르게 동작). **대신 이미 검증된 잔고조회(`getHoldings`,
`inquire-balance`)를 재사용**하는 쪽으로 선회 — 주문 전/후 해당 종목의 실제 보유수량을
직접 비교해서 체결 여부를 판단하는 방식으로 구현함. 별도 엔드포인트 스키마를 새로
검증할 필요 없이, 이미 실제 서버로 검증된 코드를 재사용한다는 장점도 있음.

**구현**: `placeMarketOrder` 직후(`paper`/`live`만) `fill_confirm_timeout_seconds`
(기본 10초) 동안 2초 간격으로 `getHoldings()`를 다시 불러서 그 종목의 수량이 기대한
값(매수면 기존+주문량, 매도면 0)에 도달했는지 확인. 타임아웃 내내 안 바뀌면 미체결로
보고 포지션/로그에 전혀 반영 안 함(매도는 원래 포지션 그대로 유지, 매수는 그냥 스킵).
중간값이면 부분체결로 그 수량만 반영. `mock`/`sim`은 애초에 동기 체결이라 이 로직
자체를 안 탐(`mode`로 분기).

## 시작 시 미체결 주문 자동 취소 (2026-07-23)

위 체결확인 기능을 만든 뒤, 재시작할 때마다 이전 실행에서 남은 미체결 주문이 계속
쌓이지 않도록 시작 시 미체결 주문을 취소하는 기능을 추가.

**1차 시도(실패)**: 정정취소가능주문조회(`inquire-psbl-rvsecncl`, "취소 가능한 주문
*목록*을 달라")를 모의투자 서버에 호출하니
`{"rt_cd":"1","msg1":"모의투자에서는 해당업무가 제공되지 않습니다."}`로 거부됨. 여기서
"모의투자는 미체결 주문을 취소할 방법이 아예 없다"고 결론 내렸다가, 사용자가
"말이 안 된다"고 지적 — 맞는 지적이었음.

**재확인**: "목록 조회"와 "취소 실행"은 서로 다른 엔드포인트. 조회(`inquire-psbl-rvsecncl`)만
막혀있는 거지 취소 자체(`order-rvsecncl`, tr_id `VTTC0803U`)가 막혀있다는 근거는 없었음.
실제로 이미 알고 있던 주문번호(`0000013603`) 하나로 직접 취소를 호출해보니
`{"rt_cd":"0","msg1":"모의투자 취소주문이 완료 되었습니다."}` — **정상 동작**.
`KRX_FWDG_ORD_ORGNO`를 빈 문자열로, `ORD_QTY=0`+`QTY_ALL_ORD_YN=Y`(잔량 전부 취소)로
넘겨도 통과됨 — 즉 주문번호만 있으면 그 외 상세정보(원주문 지점번호, 잔여수량 등)를
몰라도 취소 가능. 이 발견 덕분에 막혀있는 "목록 조회" API가 굳이 필요 없어짐.

**최종 구현**: 로컬 장부 파일(`pending_orders.txt`, 프로세스 작업 디렉토리, 주문번호
한 줄씩)로 대체 — 이 봇이 직접 낸 주문만 추적하면 충분하므로, 매수/매도 시 주문번호를
장부에 追加하고, 체결확인 폴링이 끝나면(전량체결/부분체결 잔량취소/미체결 취소 전부)
장부에서 지움. 시작 시 장부에 남아있는 것(직전 실행이 강제종료 등으로 못 지운 것)을
전부 취소 시도. `live` 계좌는 여기에 더해 실제 목록조회 API도 같이 돌려서(그건 막혀있지
않음) 장부 밖의 주문(수동 주문 등)까지 잡아냄.

**이번에 실제로 정리한 것**: `trading.log`에서 과거 주문번호 24건을 뽑아 그 중 23건
(테스트로 이미 취소한 1건 제외)을 장부에 미리 채워 넣고 봇을 재시작 → 23건 전부
취소 성공 확인(`매수가능금액`이 1,026원 → 2,968,477원으로 회복). 다만 KIS가 예약을
전부 즉시 반영하진 않는지 아직 예수금 1,000만원 전액까지는 안 돌아옴 — 시간이 좀
더 지나면 마저 회복될 수 있음, 확실친 않음.

## 1등 후보가 막히면 다른 후보로 안 넘어가던 문제 (2026-07-23)

사용자가 실제로 겪은 상황: `max_positions=20`인데 SK오션플랜트 하나만 계속 들고
있고 더 안 삼. 로그로 원인 확인: SK오션플랜트가 매 사이클 기댓값이 압도적으로 높아서
(가격이 비싸서 `gain = price × take_profit_pct`가 큼) 항상 "이번 사이클의 후보"로
뽑히는데, `max_topups_per_symbol`(3회) 상한에 걸려서 매번 "추가매수 보류"로 끝남 —
그 사이클에 다날, 아진엑스텍 등 다른 BUY 신호 종목이 같이 있어도 애초에 검토조차
안 되고 그냥 스킵됨. 원래 설계가 "사이클마다 기댓값 1등 후보 하나만 계산해서 액션"
구조라 1등이 막히면 그걸로 끝이었음.

**수정**: 스캔에서 BUY 신호가 뜨는 종목을 전부 모아서(기존엔 최고 기댓값 하나만
추적) 기댓값 내림차순으로 정렬해두고, 위에서부터 순회하면서 "실제로 살 수 있는"
첫 종목에만 액션. 막히는 이유(추가매수 횟수 상한/투입비율 상한/자금부족/교체매도
조건 불충족)가 나오면 로그 남기고 다음 후보로 넘어감(`continue`), 하나라도 성공하면
그 사이클은 종료. 교체매도(로테이션)는 사이클당 최대 1건으로 계속 제한함(후보를
순회하면서 여러 번 팔지 않도록 `evictedThisCycle` 플래그로 막음) — 후보들이 기댓값
내림차순이라 1등이 교체매도 문턱을 못 넘으면 그 뒤 후보(기댓값이 더 낮음)도 이론상
못 넘는게 당연하지만, 이미 보유 중인 종목의 추가매수 경로는 이 상한과 무관하게 계속
검토됨.

mock 모드로 `max_topups_per_symbol=0`(모든 추가매수 즉시 차단)으로 강제 재현해서
확인: 같은 사이클에 여러 종목이 BUY로 뜰 때 기댓값 높은 순으로 각각 다른 종목에
분산 매수되는 것, 막힌 후보가 "다음 후보로" 로그와 함께 정상적으로 스킵되는 것까지
확인함(`max_positions=4`로 4개 종목 전부 분산 매수 성공).

## 리스크 관리: 손절/일일 손실 한도/고정비율 리스크 사이징 (2026-07-23)

사용자가 "월가는 리스크를 어떻게 관리하나" 물어봐서 VaR/Kelly/헤지자산 등 훑어준 뒤,
그중 구현난이도 대비 효과가 큰 것부터 골라서 진행:

- **손절(`stop_loss_pct`)**: 기존 익절(`takeProfitHit`) 체크 바로 옆에
  `netPct <= -stopLossPct` 추가. 매도 사유 우선순위는 `악재 > 익절 > 손절 > 데드크로스`
  (여러 조건이 동시에 걸려도 하나만 골라 로그에 남겨야 해서 순서를 정함, 실제로 여러
  개 동시에 걸리는 경우는 드묾).
- **고정비율 리스크 사이징(`risk_per_trade_pct`)**: 월가 CTA/프롭데스크가 흔히 쓰는
  fixed-fractional risk model — "이 트레이드가 손절선에서 잘리면 총자산의 X%만 잃는다"는
  조건으로 수량을 역산(`수량 = 총자산×리스크비율 / (가격×손절폭)`). 기존
  `position_cash_fraction` 기반 사이징을 완전히 대체하지 않고, 리스크 기반 수량과
  `position_cash_fraction` 기반 수량 중 **더 작은 쪽**을 최종 채택 — 손절폭이 아주
  타이트하게 설정된 경우(예: stop_loss_pct=0.1%) 리스크 모델만으로는 지나치게 큰
  수량이 나올 수 있는데, 그런 경우에도 기존 안전판이 계속 작동하게 하려고 이렇게
  결정함(사용자에게 "둘 중 더 작은 쪽" 방식으로 갈지 물어보진 않았고, 판단해서
  결정 — 문제 생기면 조정).
- **일일 손실 한도(`daily_loss_limit_pct`, 서킷브레이커)**: 오늘 누적 실현손익
  (매도 체결 `netProfit` 합)이 총자산의 -X%를 넘으면 그날은 스캔 자체를 스킵(Phase B
  전체 건너뜀 — API 호출도 아낌). Phase A(모니터링/매도)는 계속 정상 동작해서
  손절/익절 자체가 막히진 않음. 별도 DB 없이 **`trades.log`에서 오늘 날짜분 SELL
  행의 `netProfit`을 합산해서 복원**하는 방식을 씀 — 재시작해도 같은 날이면 한도
  계산이 끊기지 않고, 새 파일이나 상태 저장 로직을 안 만들어도 됨(이미 있는
  `trades.log`를 재사용). 날짜가 바뀌면(`YYYY-MM-DD` 문자열 비교) 자동 초기화.
- mock 모드로 `stop_loss_pct=0.01`, `daily_loss_limit_pct=0.003`으로 강제 재현해서
  검증: 손절 매도가 정확한 사유로 찍히는 것, 사이징이 손계산과 정확히 일치하는 것
  (리스크예산/손절폭 vs 현금비율 중 작은 쪽 채택), 손실 한도 도달 후 스캔이 멈추고
  모니터링 메시지만 반복되는 것, 재시작 시 `trades.log`에서 오늘 손실이 정확히
  복원되는 것까지 전부 확인함.

## 알려진 한계 / 다음에 할 만한 것

- 기댓값의 "확률"은 뉴스 감성 점수를 선형 매핑한 휴리스틱일 뿐 백테스트로 보정된
  값이 아님 — 실거래(`mode: live`) 투입 전에 반드시 백테스트 필요.
- 정규장 체크는 요일+시각만 봄, 공휴일 캘린더 없음 — 공휴일엔 스캔은 돌지만 주문
  단계에서 거부됨.
- 잔고 조회(예수금/보유수량 검증) 로직 없음.
