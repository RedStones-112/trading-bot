다음에 이 프로젝트를 이어서 작업할 때(사람이든 Claude든) 여기부터 읽을 것.
`README.md`는 사용법 중심, 이 파일은 "왜 이렇게 만들었는지"와 지금까지의 경위 중심.

## 프로젝트 개요

- 목적: 한국투자증권(KIS) Developers API로 붙는 실제/모의 주식 트레이딩 봇 (C++ 초안)
- 위치: `C:\Users\wogur\OneDrive\Desktop\project\trading`
- GitHub: https://github.com/RedStones-112/trading-bot — **public repo**
- 빌드: CMake + MSYS2 UCRT64 g++. WinHTTP(Windows 내장)로 HTTPS, libcurl 등 외부
  의존성 없음. JSON은 nlohmann/json 헤더 하나만 vendor(`third_party/json.hpp`).

## 아키텍처

- `src/broker.hpp`: `IBroker` 인터페이스 + `StockInfo`(code/name/price/dayChangePct).
  구현체 3개:
  - `KisClient` (src/kis_client.*): 실제 KIS REST API
  - `MockBroker` (src/mock_broker.*): 완전 오프라인, 가상 종목 5개 랜덤워크
  - `SimBroker` (src/sim_broker.*): KIS 실시세는 그대로 쓰되 주문은 로컬 가상 체결
- `src/strategy.hpp`: 순수함수 3개 — `smaCrossSignal`(골든/데드크로스), `smaMomentum`,
  `netProfitPct`(수수료/세금 반영 순손익률)
- `src/news_crawler.hpp/cpp`: RSS 피드 파싱(직접 짠 `<item>`/`<title>` 추출, XML
  라이브러리 없음) + Google News 검색(`news.google.com/rss/search`) + 키워드 기반
  감성 점수화
- `src/main.cpp`: config.json 로드 → mode별 브로커 생성 → 정규장 체크 → (보유 없으면)
  스캔/선택/매수, (보유 중이면) 모니터링/매도 루프. `holdings.txt`를 5초마다 갱신하는
  별도 스레드도 여기서 띄움.
- `tests/test_strategy.cpp`: 결정론적 self-check (SMA 크로스, netProfitPct,
  MockBroker, NewsCrawler 감성점수). `cmake --build build --target test_strategy`로 빌드.

## 동작 흐름 (현재 최종본)

0. `paper`/`live` 모드는 평일 09:00~15:30(로컬 시각)가 아니면 스캔 자체를 스킵 (장
   마감 시 KIS가 주문을 거부하는데, 그 전에 스캔+뉴스크롤링을 다 하는 낭비를 막기 위함).
1. 보유 종목 없으면: KIS 거래량순위 API(세그먼트 `0000`/`1001`/`2001` 병합, 세그먼트당
   최대 30개 → 중복 제거 후 최대 ~90개) → 레버리지/인버스 ETF·ETN 이름 필터링 →
   `watchlist_size`만큼 후보 확보.
2. 후보가 `deep_scan_limit`보다 많으면 전일대비율(`prdy_ctrt`, 순위조회 응답에 이미
   있는 무료 필드)로 정렬해서 상위 N개만 정밀 분석(일봉 히스토리 조회는 비용이 큼).
3. RSS 뉴스(매경/한경/연합뉴스) 크롤링 → 종목명 언급 헤드라인에서 긍정/부정 키워드로
   감성 점수화.
4. 각 후보의 SMA 골든크로스 여부 확인 → Buy 신호 후보 중 기댓값(`이득 × 확률`, 이득은
   `take_profit_pct` 기준 목표수익, 확률은 감성점수를 0.05~0.95로 매핑한 휴리스틱)이
   가장 높은 종목 **하나만** 매수.
5. 보유 중엔 매 poll마다: (a) 데드크로스, (b) 수수료/세금 차감 순손익률이
   `take_profit_pct` 이상, (c) 보유종목명으로 Google News를 직접 검색한 감성점수가
   `bad_news_sentiment_threshold` 이하(악재) — 셋 중 하나만 걸려도 즉시 매도.
   (c)는 보유종목이 스캔의 거래량순위 후보에서 벗어나도 계속 추적하기 위해 스캔의
   뉴스풀과 별개로 매번 직접 검색함.
6. `holdings.txt`(5초 고정 간격) / `trades.log`(체결마다 CSV 한 줄) / 콘솔+`trading.log`
   (실시간 스캔/시그널/체결 로그).

## 사용자가 확인해준 설계 결정

- 기댓값 공식: **이득 × 확률**(표준). "이득/확률"은 채택 안 함.
- 매도 트리거: 데드크로스 **and** 수익실현 **and** 악재감지, 셋 다 사용(OR 조건).
- 뉴스 크롤링 범위: **RSS만** 우선 구현. 유튜브/SNS는 명시적으로 제외
  (공식 API 인증 필요, 비공식 스크래핑은 ToS/차단 위험 커서 이 단계에서는 안 함).
- GitHub repo는 처음 private로 만들었다가 사용자 요청으로 public 전환.

## 발견하고 고친 버그들 (시간순)

1. WinHTTP `GetLastError()`를 핸들 정리(`WinHttpCloseHandle`) *이후*에 호출해서 항상 0으로
   나오던 버그 — 실패 직후에 캡처하도록 수정.
2. WinHTTP 연결 포트를 443으로 하드코딩 — KIS는 모의투자 29443, 실서버 9443 커스텀
   포트를 씀. `http::request`에 포트 파라미터 추가.
3. 콘솔에 한글이 깨져 보임 — `SetConsoleOutputCP(CP_UTF8)` 누락. (로그 파일 자체는
   원래도 정상이었음, 콘솔 표시만 문제)
4. KIS 레이트리밋(`EGW00201`, 모의투자 계좌가 특히 엄격) — 시작 시 여러 API 호출이
   연달아 나가서 발생. 호출 사이 1.1초 pause 추가 + 일봉 히스토리는 매수 시점에
   한 번만 캐시(매 poll마다 재조회 안 함)로 호출 수 자체를 줄임.
5. `watchlist_size=100`을 줘도 10개만 반환 — 거래량순위 API가 세그먼트 하나당 최대
   30개까지만 응답하는 게 원인. 세그먼트 3개(`0000`/`1001`/`2001`) 병합+중복제거로
   해결(최대 ~90개까지 확보 가능).
6. MockBroker 회귀 버그: 순위조회 최적화(현재가를 랭킹 응답에서 재사용) 이후, mock
   모드의 랜덤워크가 `getCurrentPrice()`에서만 진행되고 `getTopVolumeStocks()`에서는
   안 움직여서 스캔 중 가격이 고정돼 있던 문제 — `advance()` 헬퍼로 공유해서 해결.

## 다중 포지션 포트폴리오로 확장 (2026-07-23)

기존엔 종목 하나만, 고정 `qty`(=1주)로만 보유했음. 사용자 요청으로 4가지 규칙 추가:
잔고 기반 운용, 여유자금 있으면 추가매수, 더 나은 후보 나오면 최저 기댓값 보유종목과
교체, 낱개매수 지양. 구현 전 Plan 서브에이전트로 설계 검증을 한 차례 거쳤고, 그
과정에서 아래 버그/설계 실수를 미리 잡았음 — 나중에 이 로직을 건드릴 때 왜 이렇게
됐는지 참고할 것.

- **로테이션 부등식 방향**: 처음 설계는 `신규후보EV > 보유종목EV - 교체비용`이었는데,
  이러면 비용이 클수록 오히려 교체가 쉬워지는 반대 방향 버그였음. 올바른 조건은
  `신규후보EV > 보유종목EV + 교체비용` — "교체가 그 비용을 감수할 만큼 나아야 한다"는
  뜻이 되려면 비용을 후보 쪽 문턱을 높이는 데 써야 함. 교체 대상(최저 EV 보유종목)
  선정 자체는 비용 반영 없이 순수 EV 최솟값으로 고름(비용을 선정에도 섞으면 비싼
  포지션이 부당하게 안 팔리는 쪽으로 편향됨).
- **평단가에 수수료 섞지 않기**: 가중평균 평단가(`avgBuyPrice`) 계산은 반드시
  체결가(fee 안 붙은 순수 가격)로 해야 함. `netProfitPct`가 fee/tax를 이미 별도로
  반영하는데 평단가에도 fee가 섞이면 익절 판정이 이중으로 틀어짐.
- **매수 수량**: `position_cash_fraction × 그 시점 매수가능금액`으로 계산하되, 그
  결과가 0주여도 실제 매수가능금액으로 1주는 살 수 있으면(비싼 종목) 1주로 폴백함 —
  안 그러면 낱개매수 지양 규칙 때문에 오히려 살 수 있는 매수 기회를 놓치는 역설이
  생김.
- **락 처리**: 스캔/시세/뉴스 조회 같은 느린 I/O는 항상 언락 상태에서 하고, 공유
  포지션 맵을 실제로 바꿀 때만 짧게 락. 매수/매도 실행 스레드는 메인 루프 하나뿐이라
  (statusWriterLoop는 읽기 전용) 동시 mutation 경합 자체는 없음.
- **broker에 fee/tax를 상태로 안 둠**: MockBroker/SimBroker가 가상 현금 잔고를
  정확히 증감하려면 fee_rate/tax_rate가 필요한데, 생성자에 박아넣지 않고
  `placeMarketOrder(..., feeRate, taxRate)` 호출 시점 인자로 받음 — KisClient는 그냥
  무시(실계좌 잔고에 이미 반영돼 있으므로).
- **`getBuyableCash()` 실제 검증**: KIS 매수가능조회(`inquire-psbl-order`,
  `ord_psbl_cash` 필드)로 구현하고, 실제 모의투자 서버에 붙여서 ~994만원(기본 1000만원
  계좌 기준)이 나오는 것까지 확인함. `mode: "live"`(실전 서버)로는 아직 검증 안 함.
- mock 모드로 짧게 돌려서 확인: 한 사이클에 8주 매수(고정 1주 아님, 규칙 4 확인),
  두 종목 동시 보유(규칙 2 확인), 같은 종목 추가매수 시 가중평균 평단가가 정확히
  계산됨(14주@60294 + 7주@60182 → 21주@60257 확인)까지 검증. 로테이션(규칙 3)은
  이 짧은 검증 실행에서는 조건이 실제로 발동하지 않아서 라이브로 직접 보진 못했고
  코드 리뷰로만 검증함 — 나중에 `max_positions`를 낮게(예: 1) 잡고 오래 돌리면
  더 쉽게 관찰 가능할 것.

## 한 종목에 계속 쪼개서 추가매수되던 문제 (2026-07-23)

실계좌 paper 모드로 돌리다가 세아메카닉스(396300) 한 종목에 728→511→358→...→10주
식으로 9분 내내 계속 추가매수만 되는 걸 발견. `trading.log`로 원인 확인:

- 그 종목 현재가가 9분 내내 정확히 4095원으로 고정(사실상 거래가 멈춘 종목으로
  보임). `smaCrossSignal`은 "저장된 과거 일봉 + 지금 이 순간의 실시간가" 두 값으로
  매 poll마다 새로 크로스 여부를 판정하는 구조라서, 가격이 크로스 임계값 위에
  머물러 있는 한(이 경우엔 가격이 아예 고정이라 계속) **매 poll 계속 BUY로 나옴** —
  "막 크로스한 순간"에만 한 번 잡히는 게 아니라 크로스 이후 계속 위에 있는 동안
  내내 잡힘. 원래(단일 포지션 시절) 코드는 보유 중이면 스캔 자체를 안 했어서 이
  문제가 안 보였는데, 이번 다중 포지션 개편으로 보유 중에도 매 poll 스캔하게 되면서
  드러남.
- 기댓값도 가격이 고정이라 매번 40.95로 동일 → 항상 최고 기댓값 후보로 뽑힘 →
  이미 보유 중이니 매번 "추가매수" 분기 → `position_cash_fraction`(0.3)이 "그 시점

## 주문 접수 ≠ 체결 -- 전량 미체결이었던 사고 (2026-07-23)

세아메카닉스 무한분할매수(위 항목)를 상한으로 막은 뒤, 사용자가 "거래내역에 아무것도

특성일 수도 있음, 문서와 다르게 동작). **대신 이미 검증된 잔고조회(`getHoldings`,

직접 비교해서 체결 여부를 판단하는 방식으로 구현함. 별도 엔드포인트 스키마를 새로
검증할 필요 없이, 이미 실제 서버로 검증된 코드를 재사용한다는 장점도 있음.

**구현**: `placeMarketOrder` 직후(`paper`/`live`만) `fill_confirm_timeout_seconds`

값(매수면 기존+주문량, 매도면 0)에 도달했는지 확인. 타임아웃 내내 안 바뀌면 미체결로

## 시작 시 미체결 주문 자동 취소 (2026-07-23)

위 체결확인 기능을 만든 뒤, 재시작할 때마다 이전 실행에서 남은 미체결 주문이 계속

"모의투자는 미체결 주문을 취소할 방법이 아예 없다"고 결론 내렸다가, 사용자가

**재확인**: "목록 조회"와 "취소 실행"은 서로 다른 엔드포인트. 조회(`inquire-psbl-rvsecncl`)만

**최종 구현**: 로컬 장부 파일(`pending_orders.txt`, 프로세스 작업 디렉토리, 주문번호
한 줄씩)로 대체 — 이 봇이 직접 낸 주문만 추적하면 충분하므로, 매수/매도 시 주문번호를

전부 취소 시도. `live` 계좌는 여기에 더해 실제 목록조회 API도 같이 돌려서(그건 막혀있지

## 리스크 관리: 손절/일일 손실 한도/고정비율 리스크 사이징 (2026-07-23)

사용자가 "월가는 리스크를 어떻게 관리하나" 물어봐서 VaR/Kelly/헤지자산 등 훑어준 뒤,
그중 구현난이도 대비 효과가 큰 것부터 골라서 진행:

- **손절(`stop_loss_pct`)**: 기존 익절(`takeProfitHit`) 체크 바로 옆에
  `netPct <= -stopLossPct` 추가. 매도 사유 우선순위는 `악재 > 익절 > 손절 > 데드크로스`
  (여러 조건이 동시에 걸려도 하나만 골라 로그에 남겨야 해서 순서를 정함, 실제로 여러
  개 동시에 걸리는 경우는 드묾).
- **고정비율 리스크 사이징(`risk_per_trade_pct`)**: 월가 CTA/프롭데스크가 흔히 쓰는
  fixed-fractional risk model — "이 트레이드가 손절선에서 잘리면 총자산의 X%만 잃는다"는
  조건으로 수량을 역산(`수량 = 총자산×리스크비율 / (가격×손절폭)`). 기존
  `position_cash_fraction` 기반 사이징을 완전히 대체하지 않고, 리스크 기반 수량과
  `position_cash_fraction` 기반 수량 중 **더 작은 쪽**을 최종 채택 — 손절폭이 아주
  타이트하게 설정된 경우(예: stop_loss_pct=0.1%) 리스크 모델만으로는 지나치게 큰
  수량이 나올 수 있는데, 그런 경우에도 기존 안전판이 계속 작동하게 하려고 이렇게
  결정함(사용자에게 "둘 중 더 작은 쪽" 방식으로 갈지 물어보진 않았고, 판단해서
  결정 — 문제 생기면 조정).
- **일일 손실 한도(`daily_loss_limit_pct`, 서킷브레이커)**: 오늘 누적 실현손익
  (매도 체결 `netProfit` 합)이 총자산의 -X%를 넘으면 그날은 스캔 자체를 스킵(Phase B
  전체 건너뜀 — API 호출도 아낌). Phase A(모니터링/매도)는 계속 정상 동작해서
  손절/익절 자체가 막히진 않음. 별도 DB 없이 **`trades.log`에서 오늘 날짜분 SELL
  행의 `netProfit`을 합산해서 복원**하는 방식을 씀 — 재시작해도 같은 날이면 한도
  계산이 끊기지 않고, 새 파일이나 상태 저장 로직을 안 만들어도 됨(이미 있는
  `trades.log`를 재사용). 날짜가 바뀌면(`YYYY-MM-DD` 문자열 비교) 자동 초기화.
- mock 모드로 `stop_loss_pct=0.01`, `daily_loss_limit_pct=0.003`으로 강제 재현해서
  검증: 손절 매도가 정확한 사유로 찍히는 것, 사이징이 손계산과 정확히 일치하는 것
  (리스크예산/손절폭 vs 현금비율 중 작은 쪽 채택), 손실 한도 도달 후 스캔이 멈추고
  모니터링 메시지만 반복되는 것, 재시작 시 `trades.log`에서 오늘 손실이 정확히
  복원되는 것까지 전부 확인함.

(2단계 항목 포함 전체 맥락).

### 핵심 설계 결정: 여러 신호를 하나의 clamp된 확률에 합치지 않음
처음엔 `probabilityFromSentiment`처럼 전부 하나의 0.05~0.95 확률에 밀어넣으려 했는데,
Plan 서브에이전트가 지적: 강한 신호 하나가 다른 신호를 집어삼키거나 클램프에서 정보가
사라져서(여러 조합이 전부 0.95로 saturate) 후보 랭킹 동점 처리가 엉망이 됨. 그래서
`probability`는 감성 전용으로 그대로 두고, 새 신호 3개는 `gain`에 곱하는 독립적인
배수(`[0.8, 1.3]` 각자 클램프)로 분리함: `기댓값 = gain × 밸류에이션배수 ×
모멘텀배수 × 관심도배수 × probability`.

### KIS API 리서치로 범위가 크게 줄어든 것들
- **PER**: 별도 순위 API가 필요할 줄 알았는데, 이미 쓰고 있는 `inquire-price`
  (FHKST01010100) 응답에 `per`/`pbr`/`eps`/`bps`가 통째로 들어있는 걸 라이브 호출로
  확인함(005930 PER 41.13, 000660 PER 32.55). 같은 응답의 `bstp_kor_isnm`(KRX 업종
  한글명)도 확인 — 005930/000660 둘 다 "전기·전자"로 나와서 동종업계 그룹핑에 실제로
  쓸 수 있는 필드임을 검증.
- **거래량 급증률**: 애초 계획은 `getDailyCloses`가 쓰는 일봉 API에 거래량(`acml_vol`)
  파싱을 추가해서 별도로 급증률을 계산하는 거였는데(그래서 `DailyBar`/`getDailyBars`
  코드까지 한 번 다 짰음), 거래량순위(`volume-rank`) 응답을 다시 보니 이미
  `vol_inrt`(전일 대비 거래량 비율, 100=변화없음) 필드가 있고 값이 `inquire-price`의
  `prdy_vrss_vol_rate`와 정확히 일치하는 걸 확인함 — **API 호출 추가 없이** 이미
  스캔에서 받고 있는 데이터로 끝나는 일이었음. `DailyBar`/`getDailyBars`는 전부
  되돌리고 `StockInfo.volumeSurgePct`(vol_inrt 파싱) 한 줄로 대체함 — 처음 설계를
  구현 중간에 라이브 데이터 보고 갈아엎은 사례.

### 이번 패스에서 구현한 것 (1단계)
- **밸류에이션 배수**: 같은 스캔 BUY 후보 중 같은 업종(`bstp_kor_isnm`)이 2개 이상
  겹치면(본인 포함 3개 이상) 동종업계 평균 PER과 비교. 미만이면 배수 1.0(미적용) —
  "평균"이 무의미할 만큼 표본이 작을 때 억지로 계산 안 함(Plan 에이전트 지적).
- **모멘텀 전이 배수**: 신규 `src/stock_tags.hpp/cpp` + `stock_tags.json` — 코드→
  태그 저장소, 처음 보는 종목은 업종 + 최근 헤드라인 co-occurrence로 휴리스틱 태그를
  자동 추정해서 저장(근사치, 검증 안 된 v1 — `parentCompany`는 신뢰성 있는 무료
  소스를 못 찾아서 빈 값으로 둠, 나중에 수동/Claude 도움으로 채우기로 함). 모멘텀은
  같은 스캔에서 오늘 급등한(전일대비율 5%+) 다른 종목과 태그가 겹치는 정도로 계산 —
  가중치 표(`tag_correlations.json`) 큐레이션은 2단계로 미루고 이번엔 단순 겹침
  개수만 씀.
- **관심도 배수**: 이번 스캔의 헤드라인 풀에서 후보명 언급 횟수(단순 카운트) +
  `vol_inrt`(거래량 급증률, 위에서 확인) — 전부 새 API 호출 없이 기존 데이터로 계산.
- **네이버뉴스**: 공식 뉴스검색 API(JSON, Client ID/Secret 무료 등록) 연동,

  남기고 완전히 스킵). Naver는 Google News RSS 같은 범용 검색 RSS가 없어서(단종됨)
  공식 API로 감. 우선 보유종목 모니터링의 종목별 검색에만 연결(Google 검색과 병합) —
  스캔 후보 전체용 일반 카테고리 풀 병합은 API 호출량 문제로 2단계로 미룸.

## 이벤트 캘린더(예정된 이벤트로 인한 등락 추정) 추가 (2026-07-24)

사용자가 요청한 5개 신호 중 남아있던 "이벤트 캘린더"(배당일/법률·정치 이벤트) 구현.
2단계로 미뤄뒀던 항목이지만 이번에 바로 요청받아 진행.

- KIS에 캘린더 API가 없고, 배당일(DART)/선거·입법 일정을 안정적으로 주는 무료 API가
  없어서(스크래핑은 이 프로젝트가 유튜브/SNS 때와 마찬가지로 피하는 영역), 새 API
  연동 대신 **수동 큐레이션 JSON**(`events.json`, `stock_tags.json`과 같은 "파일
  기반, 없으면 빈 목록" 관용)으로 구현. 신규 `src/event_calendar.hpp/cpp`
  (`ScheduledEvent`, `loadEventCalendar`, `eventMultiplier`).
- 이벤트는 `target`(종목코드 | `stock_tags.json` 태그로 업종/테마 단위 매칭 |
  `"ALL"`로 전체 후보), `date`, `impact`(부호 있는 배수 기여분)로 구성. 매칭되는
  이벤트가 `event_lookahead_days`(하드코딩 14일, 다른 배수 클램프 상수처럼 config로
  안 뺌) 이내면 날짜에 가까울수록 선형으로 크게 반영, 지난 날짜/범위 밖은 자동 제외.
  기존 3개 배수(밸류에이션/모멘텀/관심도)와 같은 자리에 `eventMult`로 추가해서
  `adjustedGain = gain × valuationMult × momentumMult × interestMult × eventMult`.
- 날짜 차이 계산은 `std::mktime`/타임존 대신 Fliegel & Van Flandern proleptic
  Gregorian day count(정수 산술)로 구현 -- DST/타임존 엣지케이스를 아예 안 만남.
- `events.json.example` 제공(형식 설명 주석 포함, config.example.json과 같은 관례).
- 검증: `tests/test_strategy.cpp`에 코드/태그/`"ALL"` 매칭, 거리 가중 감쇠, 지난
  날짜 제외, `[0.8,1.3]` 클램프, 파일 없을 때 빈 목록 반환까지 assert로 확인.
  추가로 mock 모드 격리된 디렉토리(`config.json`을 `paper`가 아닌 `mock`으로 바꾼
  임시 사본, 실계좌 config.json은 안 건드림)에서 오늘 날짜 `"ALL"` 이벤트(impact=0.2)를
  넣고 실제로 돌려서 스캔 로그의 `이벤트×1.200000`과 최종 기댓값에 정확히 반영되는
  것까지 라이브 확인.

## PER 단독 판단의 맹점 보완 -- PBR 혼합 밸류에이션 (2026-07-24)

사용자 요청: "PER이 높음에도 동종업계 주식보다 가치가 낮게 잡힌 경우"(이익 기준으로는
비싸 보여도 자산 기준으로는 저평가인 경우)에 대한 판단 로직 추가.

- 기존 밸류에이션 배수는 PER 하나만 동종업계 평균과 비교했음 -- PER이 높으면 무조건
  "비싸다"로 판정. 그런데 이익이 일시적으로 눌린 종목은 PER은 커 보여도 순자산 대비
  (PBR)로는 오히려 동종업계보다 훨씬 싸게 거래되는 경우가 있음.
- `pbr` 필드는 이미 확인된 대로(2026-07-23, `inquire-price` 응답) 새 API 호출 없이
  얻을 수 있어서 [[feedback-research-before-implementing]] 패턴 그대로 적용 --
  `Fundamentals`에 `pbr` 필드만 추가(`broker.hpp`), `kis_client.cpp`에서 한 줄
  파싱 추가.
- 2차 패스 로직을 "PER 비율 하나"에서 "PER 비율과 PBR 비율을 각각 구해서(둘 다 peer
  평균 ÷ 본인, 크면 저평가) 있는 것만 평균낸 값"으로 확장. 둘 중 하나만 유효해도
  동작(예: 적자로 PER이 의미 없는 종목은 PBR만으로 판단). `valuationMult` 이름/클램프
  범위(`[0.8, 1.3]`)는 그대로 유지.
- MockBroker 픽스처(`kMockFundamentals`)에 PBR도 추가하면서, 가상건설(000005)을
  "동종업계(전기전자) 중 PER 최고 & PBR 최저"로 일부러 설계 -- 정확히 이 기능이
  잡아야 하는 케이스를 mock 모드에서도 재현 가능하게 함.
- 검증: mock 모드로 실제 뽑힌 값(가상건설 PER=22.10/PBR=0.97, 가상전자
  PER=17.36/PBR=2.58, 가상화학 PER=13.33/PBR=3.19, 전부 이 세션의 실제 실행 로그)으로
  손계산 -- PER 비율만 보면 0.69(비싸다고 판정됐을 값)인데 PBR 비율은 2.98(강한
  저평가)이라 평균 1.84 → 클램프 상한 1.3으로 확정. 요청받은 "PER 높아도 실제로는
  저평가" 케이스가 정확히 상한 쪽으로 뒤집히는 것 확인. (같은 사이클에 세 종목이 동시에
  BUY로 뜨는 순간은 랜덤워크라 못 잡았지만, peer 비교 로직 자체는 각 종목이 실제로
  낸 PER/PBR 값으로 계산했으므로 유효한 검증.)

## KIS API 필드 카탈로그 -- 지금 쓰는 것 vs 받지만 안 쓰는 것 (2026-07-24)

사용자가 "API로 어떤 정보를 가져올 수 있는지 정리해달라"고 요청. 봇이 이미 호출 중인
5개 엔드포인트에 모의투자 계좌로 read-only GET을 날려(주문 없음, 스캐폴딩은
scratchpad에서 임시로 짜서 확인 후 폐기 -- 실키가 든 config.json 사본도 같이 지움)
raw JSON을 전부 덤프해서 필드 전체를 확인. 이 프로젝트 관행대로 문서/추측이 아니라
실제 응답으로 확인함.

- **`inquire-price`(FHKST01010100, `getCurrentPrice`/`getFundamentals`)** --
  쓰는 중: `stck_prpr`(현재가), `per`, `pbr`, `bstp_kor_isnm`(업종). 안 쓰지만 눈에
  띄는 것: `eps`/`bps`(PER/PBR 분모 원값), `frgn_hldn_qty`/`frgn_ntby_qty`/
  `hts_frgn_ehrt`(외국인 보유수량/순매수량/소진율 -- 수급 신호 후보), `w52_hgpr`/
  `w52_lwpr`(+날짜, 52주 고저 대비 위치), `hts_avls`(시가총액, 대형/소형주 필터),
  `vi_cls_code`/`mrkt_warn_cls_code`/`invt_caful_yn`(VI 발동/시장경고/투자유의 --
  리스크 필터 후보), `pvt_*`/`dmrs_val`/`dmsp_val`(피벗포인트/이격도 계열 기술적
  지지·저항값).
- **`volume-rank`(FHPST01710000, `getTopVolumeStocks`)** -- 쓰는 중: 코드/이름/
  현재가/`prdy_ctrt`(전일대비율)/`vol_inrt`(거래량비율). 안 씀: `acml_vol`/
  `acml_tr_pbmn`(누적거래량/거래대금 절대치), `nday_vol_tnrt`(N일 거래량 회전율),
  `lstn_stcn`(상장주식수).
- **`inquire-daily-itemchartprice`(FHKST03010100, `getDailyCloses`)** -- 쓰는 중:
  `stck_clpr`(종가)만. 안 씀: 그날 시가/고가/저가(캔들 전체 -- 지금은 종가만 봐서
  ATR류 변동성 지표는 못 만듦), 그날 거래량(`acml_vol`) 히스토리. 참고: 이 응답의
  `output1`에도 `per`/`pbr`/`eps`가 통째로 또 들어있음(inquire-price 없이도 얻을
  수 있었던 셈 -- 나중에 새 신호 추가할 때 참고).
- **`inquire-psbl-order`(매수가능조회, `getBuyableCash`)** -- `ord_psbl_cash`만
  씀. 안 씀: `max_buy_qty`(최대 매수가능수량, 봇이 이미 직접 계산하므로 중복).
- **`inquire-balance`(잔고조회, `getHoldings`)** -- `hldg_qty`/`prdt_name`/
  `pchs_avg_pric`만 씀. 안 씀: `output1`의 평가손익(봇이 직접 계산해서 중복),
  `output2`(계좌 총괄 -- 예수금 `dnca_tot_amt`, 총평가금액 `tot_evlu_amt` 등 --
  일일 손실 한도는 `trades.log` 재집계 방식으로 이미 별도 구현해서 지금은 안 씀).

가장 유망한 미사용 후보는 **외국인 수급**(순매수/소진율)과 **VI/시장경고 코드**(리스크
필터) -- 둘 다 새 API 호출 추가 없이 `inquire-price` 파싱 한 줄만 늘리면 씀.
아직 이 프로젝트가 호출해본 적 없는 다른 KIS 엔드포인트(재무제표, 배당, 투자자별
매매동향 등)는 이번에 확인 안 함 -- 필요해지면 그때 실제 호출로 새로 검증할 것
([[feedback-research-before-implementing]]).

## 뉴스 감성 확률 → 기술적 신호(매물대/추세/거래량)로 전면 교체 (2026-07-24)

정보인 경우가 많아서 그 때문에 손실만 나고 있다"고 지적 -- 구현 전 두 가지 범위를

   비중으로 추측했으나 사용자가 정정: **실제 매물(거래량)이 지금 가격보다 위/아래

  `getDailyBars`로 교체(`KisClient`/`MockBroker`/`SimBroker` 전부). KIS
  `inquire-daily-itemchartprice` 응답에 `stck_hgpr`/`stck_lwpr`/`acml_vol`이 이미
  같이 들어있어서(직전 턴의 API 필드 카탈로그에서 이미 "안 씀"으로 분류해뒀던 바로 그
  필드들) 새 API 호출 없이 SMA/매물대 둘 다 해결함([[feedback-research-before-implementing]]).
- `strategy.hpp`에 두 순수함수 추가(테스트 가능하도록 main.cpp static 대신 여기로):
  - `belowPriceVolumeRatio(bars, currentPrice)`: 최근 일봉들의 거래량을 그 날의
    저가~고가 구간에 균등 분배한다고 근사(분봉 없음, ponytail 코멘트로 명시)해서,
    지금 가격보다 **낮은** 가격대에 실제로 거래된 물량 비율을 0~1로 계산(0.5=중립).
    비율이 높을수록 최근 거래가 지금 가격 아래에 몰려있다는 뜻 → 위쪽 매물대(저항)가
    가볍다고 판단.
  - `probabilityFromTechnicals(volumeSurgePct, trendPct, belowRatio)`: 매물대 위치가
    가장 크게 반영(사용자가 가장 중요하게 지목한 신호), 최근 구간 추세/거래량
    급증률(기존에 이미 있던 `vol_inrt`)은 보조 신호. 기존 `probabilityFromSentiment`와
    같은 `[0.05, 0.95]` 클램프 형태 유지.
- `probabilityFromSentiment`, `bad_news_sentiment_threshold`, 보유종목 Google/네이버
  뉴스 검색(Phase A의 즉시 악재 매도용) 전부 제거. 매도 사유는 `악재>익절>손절>데드크로스`
  4개에서 `익절>손절>데드크로스` 3개로 줄어듦. 뉴스 자체(RSS/Naver)는 관심도 배수
  (언급횟수)/모멘텀 태그 자동추정에는 계속 씀 -- 이번 교체 범위는 "매수 확률"과
  "보유종목 즉시매도 트리거" 딱 두 곳으로 한정(사용자와 확인).
- `Position`에 `baseBars` 추가(매수/포지션복원 시점 캐시, `baseCloses`와 같은 방식 --
  매 poll 재조회 안 함). 보유종목 모니터링(Phase A)은 거래량 급증률 데이터가 없어서
  (스캔 랭킹을 재조회하지 않음) 중립값 100.0으로 고정하고 매물대/추세만 반영.
- (사용자 추가 요청) 일일 손실 한도 서킷브레이커를 끄고 켤 수 있게 `daily_loss_limit_enabled`
  (기본 true) config 추가.

## 알려진 한계 / 다음에 할 만한 것

- 기댓값의 "확률"은 매물대/추세/거래량급증률을 조합한 휴리스틱일 뿐(2026-07-24부터,
  이전엔 뉴스감성) 백테스트로 보정된 값이 아님 — 실거래(`mode: live`) 투입 전에
  반드시 백테스트 필요.
- 매물대(volume profile)는 분봉이 아니라 일봉의 저가~고가에 거래량을 균등 분배한
  근사치 -- 실제 체결가 분포와 다를 수 있음. 더 정밀하게 하려면 KIS 분봉 API로 교체.
- 정규장 체크는 요일+시각만 봄, 공휴일 캘린더 없음 — 공휴일엔 스캔은 돌지만 주문
  단계에서 거부됨.
- 잔고 조회(예수금/보유수량 검증) 로직 없음.
- ETF/ETN 필터는 이름 문자열에 `KODEX`/`TIGER`/`레버리지`/`인버스` 등이 포함되는지
  보는 휴리스틱 — 완벽하지 않음.
- 매수 후 모니터링용 일봉 히스토리는 매수 시점에 한 번만 캐시 — 포지션을 자정 넘어
  여러 날 들고 있으면 하루치 갱신이 안 됨(알려진 한계, README에도 명시).
- tr_id/FID 파라미터/필드명은 실제 라이브 호출로 검증했지만 KIS API가 바뀔 수 있으니
  주기적으로 재확인 권장: https://apiportal.koreainvestment.com
- `mode: "live"`로 전환하면 진짜 돈이 나감 — 충분히 검증 후에만 전환할 것.
